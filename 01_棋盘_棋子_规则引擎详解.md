# 棋盘、棋子与规则引擎详解

这份文档专门解释一个问题：

> 中国象棋最底层的规则，是怎样在这份 C++ 工程里实现出来的？

如果你把这一份真正理解了，答辩时讲“规则正确性”和“面向对象设计”就会非常顺。

---

## 一、先从整体架构理解

当前工程的核心思想，不是把所有逻辑都写在一个大函数里，而是分层：

1. `Board` 类负责“棋盘本身”
2. `ChessPiece` 抽象基类和七个派生类负责“每个棋子的走法规则”
3. `GameSession` 负责“谁走、历史记录、胜负、计时、悔棋、重复局面”
4. `MoveParser` 负责“把人输入的文字翻译成走法”
5. `EasyXApp` / `ConsoleApp` 负责“界面和交互”
6. `SearchEngine` 负责“AI 和 Hint”

也就是说，规则并不是写在界面里，而是写在引擎里。界面只是调用规则。

这是一种很重要的工程思想：

- 界面可以变
- 联机可以加
- AI 可以换
- 但规则引擎最好只有一套

## 二、棋盘在代码里是怎么表示的

### 1. 棋盘配置不是写死的，而是由 `BoardConfig` 描述

代码位置：

- `src/common/Types.h`
- `src/engine/ChessEngine.cpp`

项目中有一个 `BoardConfig` 结构体，用来描述棋盘规则参数，例如：

- 总行数 `rows`
- 总列数 `cols`
- 河界位置 `river_split_top_row`
- 九宫的行列边界
- 文件列字符，比如 `abcdefghi`
- 初始摆子表 `initial_pieces`

这样设计的好处是：

- 标准模式 `9x10` 可以用一套配置
- 扩展模式 `11x10` 也可以用另一套配置

也就是说，程序并不是把“9 列”写死在所有地方，而是尽量通过配置决定。

### 2. 棋盘内部使用的是“一维数组映射二维坐标”

代码位置：

- `src/engine/ChessEngine.h`
- `src/engine/ChessEngine.cpp`

虽然我们平时习惯把棋盘理解成二维表格，但这个项目内部使用的是：

```cpp
std::vector<std::unique_ptr<ChessPiece>> cells_;
```

它本质上是一条线性的数组，只不过通过一个 `index(position)` 函数，把二维坐标转换成一维下标。

公式可以理解成：

```text
index = row * cols + col
```

例如标准棋盘中：

- `(0,0)` 对应第 `0` 个格子
- `(0,1)` 对应第 `1` 个格子
- `(1,0)` 对应第 `9` 个格子

这种写法比二维动态数组更统一，适配 `9x10` 和 `11x10` 也更方便。

### 3. 坐标系约定

这套工程和你的课程要求基本一致：

- 左上角是 `(0, 0)`
- 右下角是标准模式下的 `(9, 8)`
- `row` 表示行
- `col` 表示列

标准模式：

- 行：`0 ~ 9`
- 列：`0 ~ 8`

扩展模式：

- 行：`0 ~ 9`
- 列：`0 ~ 10`

### 4. 河界与九宫

项目里对这些特殊区域都不是口头约定，而是通过函数判断：

- `Board::isPalaceCell(...)`
- `BoardConfig` 中的 `river_split_top_row`

标准模式下：

- 河界位于 `row = 4` 和 `row = 5` 之间
- 上方九宫：`row 0~2, col 3~5`
- 下方九宫：`row 7~9, col 3~5`

扩展模式下：

- 河界仍然在中间
- 九宫改为居中 `col 4~6`

所以你答辩时可以说：

> 我没有把九宫和河界硬编码进每个棋子函数，而是交给 `BoardConfig` 和 `Board::isPalaceCell()` 来统一判断。

这句话很加分。

## 三、为什么要用抽象基类 `ChessPiece`

代码位置：

- `src/engine/ChessEngine.h`

课程要求里明确提到要使用继承和多态。这个项目就是这样做的：

- 抽象基类：`ChessPiece`
- 派生类：
  - `KingPiece`
  - `AdvisorPiece`
  - `ElephantPiece`
  - `KnightPiece`
  - `RookPiece`
  - `CannonPiece`
  - `PawnPiece`

### 1. 抽象基类的作用

所有棋子都共有一些共同属性：

- 它属于哪一方 `side()`
- 它是什么棋子 `type()`
- 它的显示符号 `symbol()`
- 它的价值 `value()`
- 它能不能走某一步 `isPseudoLegal(...)`
- 它能生成哪些伪合法目标点 `generatePseudoLegalMoves(...)`

这些共同接口由 `ChessPiece` 统一规定。

### 2. 多态的意义

当 `Board` 遍历棋盘时，它不需要知道这个格子里是车、马、炮还是兵。

它只需要拿到一个 `ChessPiece*`，然后调用：

```cpp
piece->generatePseudoLegalMoves(...)
```

真正怎么走，由派生类自己决定。

这就是多态。

### 3. 为什么这比 `if-else` 更好

如果不用类设计，常见写法会变成：

```text
如果是车就怎样
如果是马就怎样
如果是炮就怎样
...
```

这样代码会非常长，而且很难维护。

用继承和多态后：

- 每种棋子的规则都写在自己的类里
- 改马的规则时，不会影响车的代码
- 扩展功能时更容易维护

## 四、模板 `Position<T>` 是怎么用的

代码位置：

- `src/common/Types.h`

课程要求里提到模板，这个项目用的是：

```cpp
template <typename T>
struct Position
```

它的作用就是表示棋盘坐标。

比如：

- `Position<int>{6, 0}` 表示红兵初始位置
- `Position<int>{0, 4}` 表示黑将所在的位置

### 为什么要写成模板而不是死写 `int`

因为模板的思想是“同一种逻辑，适配不同类型”。

当前项目主要使用 `Position<int>`，但你答辩时可以这样讲：

> 这里使用模板是为了体现坐标抽象能力。当前主要实例化为 `Position<int>`，但未来也可以扩展为其他坐标类型。

### 模板类还重载了比较运算符

`Position<T>` 实现了：

- `operator==`
- `operator!=`

所以程序可以很自然地判断：

- 起点和终点是否相同
- 某个目标点是否等于某个位置

这也正好对上课程要求里的“条件运算符重载”。

## 五、每种棋子的移动逻辑是怎么实现的

下面这一部分，是答辩最容易被老师问的。

建议你不要死背代码，而是背“规则 + 在代码中用什么函数实现”。

### 1. 帅 / 将

规则：

- 只能上下左右走一步
- 必须留在九宫内
- 还要避免飞将

代码实现思路：

1. `KingPiece::isPseudoLegal(...)`
   检查曼哈顿距离是否为 1
2. `Board::isPalaceCell(...)`
   检查目标点是否在己方九宫内
3. `Board::hasFlyingKings()`
   检查落子后双方将帅是否直接照面
4. `Board::isInCheck(...)`
   检查己方是否处于被攻击状态

你答辩时可以总结为：

> 将帅的规则不是只检查“走一步”，还必须配合九宫限制和飞将检测一起判断。

### 2. 仕 / 士

规则：

- 斜着走一步
- 不能出九宫

代码实现：

- `AdvisorPiece::isPseudoLegal(...)`
- `Board::isPalaceCell(...)`

### 3. 相 / 象

规则：

- 斜着走两步
- 不能过河
- 中间不能有子，也就是“塞象眼”

代码实现：

- `ElephantPiece::isPseudoLegal(...)`
- 中心点坐标：
  - `eye = ((r1+r2)/2, (c1+c2)/2)`
- 如果 `board.pieceAt(eye) != nullptr`，说明被堵住

### 4. 马

规则：

- 走“日”字
- 要检查马腿是否被堵

代码实现：

- `KnightPiece::isPseudoLegal(...)`

如果是纵向走两格：

- 检查中间那条“腿”是否有棋子

如果是横向走两格：

- 检查中间横向那条“腿”是否有棋子

这正对应课程要求中的“蹩马腿”。

### 5. 车

规则：

- 横平竖直
- 中间不能有子

代码实现：

- `RookPiece::isPseudoLegal(...)`
- `Board::countObstacles(from, to)`

只要 `countObstacles(...) == 0`，说明路径干净。

### 6. 炮

规则：

- 不吃子时，像车一样，中间必须 0 个子
- 吃子时，必须隔 1 个子，而且终点必须是敌方棋子

代码实现：

- `CannonPiece::isPseudoLegal(...)`

核心判断是：

- 目标为空：`obstacles == 0`
- 目标是敌方棋子：`obstacles == 1`

### 7. 兵 / 卒

规则：

- 永远不能后退
- 过河前只能向前
- 过河后可以向前、向左、向右

代码实现：

- `PawnPiece::isPseudoLegal(...)`

红方前进方向是行号减小，黑方前进方向是行号增大。

这是一个非常适合答辩时讲的点，因为它体现了“方向性规则”。

## 六、什么叫伪合法走法和绝对合法走法

这是中国象棋程序里最重要的概念之一。

### 1. 伪合法走法

伪合法走法的意思是：

> 只看这枚棋子自己的运动规则，这一步像不像它能走。

比如：

- 马走日字
- 炮隔山打子
- 相不能过河

这些都属于“伪合法性”。

在代码中主要对应：

- `ChessPiece::isPseudoLegal(...)`
- `ChessPiece::generatePseudoLegalMoves(...)`
- `Board::generatePseudoLegalMoves(...)`

### 2. 绝对合法走法

绝对合法走法比伪合法多了一层判断：

> 这步走完之后，自己这边的将帅不能暴露在对方攻击下。

也就是说：

- 不能自杀式送将
- 不能让将帅和对方飞将照面

在代码中，对应的是：

- `Board::generateLegalMoves(...)`

它的实现思路是：

1. 先生成伪合法走法
2. 拿当前棋盘克隆一个副本
3. 在副本上模拟这一步
4. 再检查：
   - 是否飞将
   - 己方是否被将
5. 只有通过检查，才算真正合法

这部分你答辩时一定要会讲，因为老师很爱问：

> 你是怎么避免玩家走出“自己送将”的非法步的？

标准回答就是：

> 先生成伪合法走法，再模拟落子做绝对合法性校验。

## 七、将军、将死、困毙在代码里怎么判

### 1. 将军

将军的本质是：

> 己方落子后，对方将帅处于被攻击状态。

代码依赖：

- `Board::isSquareAttacked(...)`
- `Board::isInCheck(...)`

### 2. 将死

将死的条件是：

1. 对方已经被将军
2. 对方没有任何合法走法可以脱将

代码位置：

- `GameSession::evaluateGameResultAfterMove(...)`

逻辑顺序：

1. 当前走方落子
2. 检查对方是否被将军
3. 生成对方全部合法走法
4. 如果一个都没有，就说明结束

### 3. 困毙

困毙指的是：

> 轮到某一方走，但它没被将军，同时也没有任何合法走法。

在当前实现里，`evaluateGameResultAfterMove(...)` 用“对方无合法走法”作为失败条件统一处理。

从答辩角度来说，你可以这样解释：

> 项目通过“是否被将军”和“是否还有合法步”这两个条件组合区分将死与困毙。

## 八、飞将是怎么判定的

代码位置：

- `Board::hasFlyingKings()`

飞将是中国象棋非常有代表性的规则。

判定方法非常直接：

1. 找到红帅和黑将的位置
2. 看两者是否在同一列
3. 如果在同一列，再检查中间是否有棋子
4. 如果中间没有棋子，就叫飞将

这个规则在项目中不只是“显示提醒”，而是参与合法性判断。

也就是说：

- 走出飞将状态的步，是非法步

这点很关键。

## 九、长将和循环局面怎么处理

课程要求里说：

- 长将 / 循环局面可判和
- 长将和长捉需要在重复局面中处理

当前工程做法是：

1. 基础层面保留“三次重复局面判和”
2. 进一步补充了“明显长将禁着”

共享长将辅助结构写在：

- `src/common/Repetition.h`

实战落子层检查写在：

- `GameSession::wouldViolatePerpetualCheck(...)`

所以你答辩时可以诚实地说：

> 项目已经实现了重复局面检测、长将禁着，并补充了循环追吃检测。也就是说，如果一方在重复局面中持续将军或持续追吃非将帅棋子，程序会在落子前拒绝这类走法。

这属于课程项目范围内比较完整的循环局面处理。

## 十、构造函数、复制构造、析构为什么重要

课程要求里特别强调：

- 构造函数
- 复制构造函数
- 析构函数
- 动态内存管理

当前工程中最典型的例子就是 `Board`。

### 1. 为什么 `Board` 需要复制构造

因为在生成合法走法时，会临时克隆棋盘：

- 当前局面不变
- 在副本上试走

所以 `Board` 需要实现深拷贝。

### 2. 怎么实现深拷贝

棋盘内部存的是：

```cpp
std::unique_ptr<ChessPiece>
```

这意味着不能直接浅拷贝，否则会发生所有权冲突。

所以工程里做法是：

- 每个棋子实现 `clone()`
- `Board` 复制构造时逐格调用 `clone()`

### 3. 析构为什么不用手写很多 `delete`

因为用了 `std::unique_ptr`。

这就是 RAII 思想：

- 对象离开作用域
- 智能指针自动释放内存
- 减少内存泄漏风险

答辩时你可以这样说：

> 项目虽然涉及动态分配棋子对象，但通过 `unique_ptr + clone` 的方式完成所有权管理和深拷贝，避免了裸指针带来的内存泄漏问题。

## 十一、运算符重载是怎么落地的

课程要求中的几个运算符，这个项目都能对应到具体代码。

### 1. `==` 和 `!=`

位置：

- `Position<T>` 中实现

作用：

- 比较两个坐标是否相同

### 2. `<` 和 `>`

位置：

- `ChessPiece` 中实现

作用：

- 按棋子价值比较强弱

### 3. `+`

位置：

- `Board::operator+(const Move&)`

作用：

- 返回“执行一步移动后的新棋盘”

### 4. `-`

位置：

- `Board::operator-(const Move&)`

作用：

- 返回“撤销一步之后的新棋盘”

### 5. `<<`

位置：

- `operator<<(std::ostream&, const Board&)`

作用：

- 以字符棋盘形式输出棋盘内容

### 6. `>>`

位置：

- `operator>>(std::istream&, ParsedCommand&)`

作用：

- 从输入流读入命令字符串

需要注意：

这个 `>>` 本身只负责读字符串，真正的走法解析主要是 `MoveParser::parse(...)` 完成的。

答辩时可以这样说：

> 流运算符负责接入输入输出流，而语义级解析由 `MoveParser` 专门完成，这样职责更清晰。

## 十二、这一层最适合答辩时怎么总结

如果老师问你：

> 你的规则引擎是怎么设计的？

你可以这样答：

> 我把棋盘抽象成 `Board`，把棋子抽象成 `ChessPiece` 及其七个派生类。每种棋子只负责自己的伪合法走法，`Board` 负责障碍统计、飞将检测、将军检测和绝对合法性校验。然后 `GameSession` 再在此基础上管理回合、历史记录和胜负判定。这样规则层、状态层和界面层是分离的，便于扩展到 EasyX、联机和 AI。

这一段非常适合直接背下来。
